# Changelog 生成指南

## 目标

确保每个版本的发布说明做到：
- 先讲用户可感知价值，再讲技术细节
- 版本重点清晰、可扫描
- `CHANGELOG.md` 与 GitHub Release 文案完全一致

## 为什么以前容易写偏

1. 只写“做了什么实现”，没写“用户得到什么能力”
2. 修复项没说明“问题原因 -> 修复结果”
3. 核心特性与次要技术项混在一层，重点不突出
4. CHANGELOG 与 Release 分别手写，容易不一致

## 标准结构（必须遵守）

每个版本段落按以下顺序：

1. `### 功能特性`
2. `### 其他改进`

其中：
- `功能特性` 使用编号列表（1,2,3...）
- 每条必须是“能力 + 场景 + 结果”表达
- 修复类条目必须写出因果：`修复了 X 导致的 Y，结果 Z`
- 技术实现、测试补充、文档补充放在 `其他改进`

## 推荐写法模板

```md
## [x.y.z] - YYYY-MM-DD

### 功能特性
1. **能力标题**
   - 场景：
   - 结果：
2. **修复标题**
   - 问题：修复了 X 导致的 Y
   - 结果：Z

### 其他改进
- 技术实现补充（命令、模块、重构）
- 测试覆盖补充（BDD / 集成 / E2E）
- 文档补充
```

## 发布执行流程（文案部分）

1. 先在 `CHANGELOG.md` 写对应版本段落
2. 从 `CHANGELOG.md` 提取对应版本段落作为 release notes
3. 用提取结果更新 GitHub Release 文案（不手写第二份）

## 发布前检查清单

- [ ] 当前版本段落以 `功能特性` 开头
- [ ] 每个功能点都可被非开发角色理解
- [ ] 每个修复项包含“原因 + 结果”
- [ ] `CHANGELOG.md` 与 GitHub Release 文案一致

## 本项目当前推荐的主轴（示例）

- 支持更多编程工具环境，并修复切换异常
- 支持自动修改并保存
- 支持批量修改并保存
- 完善 Skills 面板能力
- CLI 新增死链修复能力
